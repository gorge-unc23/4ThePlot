# 4ThePlot Non-Functional Requirements

| Req# | Requirement | Comments | Priority | Date | Reviewed/Approved |
| :--- | :--- | :--- | :---: | :---: | :--- |
| NFR_01 | Authenticated API actions shall show loading, success, empty, or error states and respect the app request timeout. | Applies to login, event loading, comments, registration, settings, admin pages, and upload flows. | 1 | 26/05/2026 | Ronald Lamani / Angjelina Zenunaj |
| NFR_02 | The Flutter interface shall remain usable on mobile screens with readable text, stable navigation, and clear empty/error states. | Main navigation, settings, event details, calendar, map, and admin screens must remain scannable and tappable. | 1 | 26/05/2026 | Ilion Elezaj / Gëzime Mirku |
| NFR_03 | Role-based access shall separate Goer, Business, and Admin navigation and visible actions. | Goer users see discovery-first flows, Business users see hosted-event flows, and Admin users see moderation and operations flows. | 1 | 26/05/2026 | Klaus Saliaj / Ronald Lamani |
| NFR_04 | Account passwords shall be hashed on the backend and session tokens shall be stored securely on the client. | The backend hashes passwords and the Flutter app stores authentication tokens using secure storage. | 1 | 26/05/2026 | Revi Beja / Briana Llapaj |
| NFR_05 | Forms shall validate required fields and preserve user-entered data when requests fail. | Applies to signup, login, profile editing, add/edit event, reports, host verification, admin reasons, notifications, and dispute decisions. | 1 | 26/05/2026 | Angjelina Zenunaj / Klaus Saliaj |
| NFR_06 | Image and PDF uploads shall restrict supported file types and show user-visible failure messages. | Event/profile images use image upload controls; host verification documents accept PDF files. | 1 | 26/05/2026 | Ilion Elezaj / Revi Beja |
| NFR_07 | Location-based discovery shall handle denied permissions and unavailable location services gracefully. | If location is unavailable, the Discover screen must still allow trending and search flows. | 2 | 26/05/2026 | Briana Llapaj / Ronald Lamani |
| NFR_08 | Event list loading shall use available fallback behavior where implemented and expose retry states on failure. | Discovery, hosted events, calendar, map, and admin event screens should show clear recovery options instead of blocking navigation. | 2 | 26/05/2026 | Gëzime Mirku / Angjelina Zenunaj |
| NFR_09 | Capacity-sensitive registration changes shall remain consistent and avoid visible duplicate joins. | Join and unregister actions must coordinate with capacity counts and disable repeated taps while requests are active. | 1 | 26/05/2026 | Klaus Saliaj / Revi Beja |
| NFR_10 | Admin moderation, verification, dispute, notification, and delete actions shall be audit logged. | Admin reason dialogs and backend audit records support traceability for critical operational actions. | 1 | 26/05/2026 | Ronald Lamani / Briana Llapaj |
| NFR_11 | Sensitive admin-only data shall require admin authorization. | Verification documents, disputes, chat evidence, audit logs, metrics, reports, and moderation controls must not be exposed to non-admin users. | 1 | 26/05/2026 | Angjelina Zenunaj / Briana Llapaj |
| NFR_12 | Static checks or syntax checks shall pass for touched frontend and backend files before delivery. | At minimum, edited Python files should compile and edited Dart files should be analyzer-clean or documented with known pre-existing issues. | 2 | 26/05/2026 | Ilion Elezaj / Klaus Saliaj |
